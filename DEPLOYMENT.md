# Deployment Guide for AWS EC2 (Free Tier)

This guide helps you deploy the backend to an AWS EC2 `t3.small` or `t3.micro` instance using Docker and Docker Compose.

## Prerequisites
- AWS Account with EC2 instance (Ubuntu 24.04 LTS)
- Security Group configured to allow:
  - SSH (Port 22)
  - HTTP (Port 80)
  - HTTPS (Port 443) - optional

## Quick Deploy (Recommended)

### Step 1: SSH into your EC2 instance
```bash
ssh -i your-key.pem ubuntu@your-instance-ip
```

### Step 2: Clone the repository
```bash
cd /home/ubuntu
git clone https://github.com/YOUR_USERNAME/HostelManager.git
cd HostelManager
```

### Step 3: Run the automated deployment script
```bash
chmod +x deploy.sh
bash deploy.sh
```

The script will:
- Update system packages
- Verify Docker installation
- Create production `.env` file with secure JWT secret
- Build and start all services (MongoDB + Redis + API + Nginx)
- Verify all containers are running
- Test the API health endpoint

### Step 4: Access your API
```
http://your-instance-ip/api/v1/health
```

## Manual Deployment (if script fails)

### Step 1: Update system and install Docker
```bash
sudo apt update
sudo apt upgrade -y
sudo apt install -y docker.io docker-compose git

# Start Docker service
sudo systemctl start docker
sudo systemctl enable docker

# Add ubuntu user to docker group
sudo usermod -aG docker ubuntu
```

### Step 2: Clone and configure
```bash
cd /home/ubuntu
git clone https://github.com/YOUR_USERNAME/HostelManager.git
cd HostelManager

# Create .env file
cp api/.env.example api/.env
nano api/.env
```

Update these fields in `.env`:
- `ALLOWED_ORIGINS`: Your instance IP or domain
- `JWT_SECRET`: Keep the default (generated by deploy.sh) or change to custom value
- `RAZORPAY_*`: Add if you have Razorpay keys
- `ZEPTO_MAIL_API_KEY`: Add if you have Zoho key

### Step 3: Start services
```bash
sudo docker-compose up -d
```

### Step 4: Monitor if everything starts
```bash
# Check all containers
sudo docker-compose ps

# View logs
sudo docker-compose logs -f

# Test API
curl http://localhost:8000/api/v1/health
```

## Troubleshooting

### Services won't start - RAM issue
On `t3.small` (2GB RAM), MongoDB may fail. Solution:

**Option A**: Use MongoDB Atlas (cloud) instead
```bash
# Get free MongoDB cluster at https://www.mongodb.com/cloud/atlas
# Update MONGO_URL in .env:
MONGO_URL=mongodb+srv://username:password@cluster.mongodb.net/hostelmanager
```

**Option B**: Upgrade to `t3.medium` (4GB RAM)

### Nginx showing 502 error
Backend container may still be starting. Wait 30 seconds and retry:
```bash
# Check backend logs
sudo docker-compose logs backend

# Restart just the backend
sudo docker-compose restart backend
```

### Can't access API from outside
Check security group inbound rules:
- Port 80 (HTTP) must allow traffic from 0.0.0.0/0
- Port 22 (SSH) should allow your IP only

sudo docker-compose logs -f backend

## Production Notes

- **JWT_SECRET**: Should be a long random string. The deploy script generates one automatically.
- **ALLOWED_ORIGINS**: Make sure to add your frontend domain(s) and your API domain
- **ENFORCE_HTTPS**: Set to `true` in production (requires SSL certificate)
- **ENV**: Set to `production` for production deployments
- **Backups**: MongoDB data is stored in `mongo-data` volume. For production, set up automated backups.

```bash
sudo apt-get install certbot python3-certbot-nginx -y
# Note: You need a domain name pointing to your EC2 IP
sudo certbot --nginx -d yourdomain.com
```

## 6. Performance Optimization for Free Tier
- **SWAP Space:** t2.micro has only 1GB RAM. It's highly recommended to add 2GB of swap space to prevent the server from crashing.
```bash
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
```

- **Database:** Avoid running MongoDB locally on the same instance if possible; use MongoDB Atlas.
- **Workers:** Keep the number of Gunicorn workers low (set to 2 in the Dockerfile).
